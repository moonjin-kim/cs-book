# 객체지향

객체지향 면접은 **객체에 책임을 나누고, 변경에 강한 구조를 만들기 위해 캡슐화·추상화·다형성·의존성 관리를 어떻게 사용하는지**를 설명하는 것이 핵심입니다.

## 핵심 섹션 맵

| 우선순위 | 섹션 | 먼저 답할 수 있어야 하는 질문 |
| --- | --- | --- |
| 1 | OOP 4대 특징 | 캡슐화·추상화·다형성·상속의 목적을 설명하고, 재사용을 위해 상속보다 조합을 먼저 검토하는 이유를 말할 수 있는가? |
| 2 | TDA: Tell, Don't Ask | 객체의 데이터를 꺼내 판단하는 코드를 객체에게 행동을 시키는 코드로 바꿔야 하는 이유를 설명할 수 있는가? |
| 3 | SOLID | SRP부터 DIP까지 다섯 원칙의 의미와, OCP를 언제 적용하는 것이 현실적인지 말할 수 있는가? |
| 4 | 추상 클래스 vs 인터페이스 | 공통 상태/로직 재사용과 역할/계약 정의 중 어느 상황에 무엇을 쓸지 구분할 수 있는가? |
| 5 | 오버로딩 vs 오버라이딩 | 결정 시점(컴파일 타임 vs 런타임)과 판단 기준의 차이를 설명할 수 있는가? |
| 6 | 응집도와 결합도 | 높은 응집도·낮은 결합도가 왜 좋은 설계인지 책임과 변경 관점으로 설명할 수 있는가? |
| 7 | DI | 생성자·Setter·Method 주입을 구분하고, 필수 의존성에 생성자 주입을 선호하는 이유를 말할 수 있는가? |
| 8 | 디자인 패턴 | Singleton·Strategy·Template Method 등이 각각 어떤 반복 설계 문제를 해결하는지 설명할 수 있는가? |
| 9 | 방어적 복사 | 가변 객체를 언제 복사해야 하고, 컬렉션만 복사할 때 원소가 공유되는 한계를 설명할 수 있는가? |
| 10 | 함수형 프로그래밍 | 순수 함수와 불변 데이터가 객체지향과 어떻게 상호 보완되는지 말할 수 있는가? |
| 11 | 실전 면접 Q&A | 짧은 답변으로 OOP 개념의 선택 이유와 트레이드오프를 방어할 수 있는가? |

## 빠른 요약

| 주제 | 핵심 답변 |
| --- | --- |
| 캡슐화 | 변경 가능성을 객체 내부로 숨겨 외부 영향 범위를 줄입니다. |
| 추상화 | 중요한 역할과 계약만 드러내고 세부 구현을 감춥니다. |
| 다형성 | 같은 메시지를 객체마다 다른 방식으로 처리합니다. |
| SOLID | 변경 이유와 의존성을 관리하기 위한 설계 원칙입니다. |
| DI | 필요한 의존 객체를 외부에서 받아 결합도를 낮춥니다. |
| 응집도/결합도 | 관련 책임은 모으고, 모듈 간 의존은 줄입니다. |
| 디자인 패턴 | 반복되는 설계 문제의 이름 붙은 해법입니다. |

## 면접 답변 프레임

객체지향 질문은 용어 암기보다 **어떤 문제를 어떤 설계 결정으로 풀었는지**를 설명하는 것이 중요합니다.

1. 정의: 개념이 무엇인지 한 문장으로 정리합니다. 예를 들어 캡슐화는 변경 가능성을 객체 내부로 숨기는 것입니다.
2. 해결하려는 문제: 이 개념이 없으면 어떤 어려움이 생기는지 말합니다. 데이터를 외부에서 직접 다루면 변경 영향 범위가 넓어집니다.
3. 원칙/패턴 적용: SOLID, TDA, 디자인 패턴 중 무엇으로 문제를 다루는지 연결합니다. 조건 분기가 늘면 Strategy, 상태 모순을 막아야 하면 Template Method를 검토합니다.
4. 트레이드오프: 얻는 것과 치르는 비용을 함께 말합니다. 책임을 나누면 결합도는 낮아지지만 클래스 수와 초기 설계 비용은 늘어납니다.
5. 실무 예: 응집도·결합도 조정, 의존성 주입, 방어적 복사처럼 실제 코드에서 어떻게 드러나는지 한 가지 예를 듭니다.

## OOP 4대 특징

| 특징 | 의미 |
| --- | --- |
| 캡슐화 | 데이터와 행위를 묶고 내부 구현을 숨김 |
| 추상화 | 핵심 개념과 계약만 드러냄 |
| 다형성 | 같은 인터페이스가 여러 구현으로 동작 |
| 상속 | 상위 타입의 속성과 동작을 재사용/확장 |

주의:

- 상속은 강한 결합을 만들 수 있습니다.
- 재사용만을 위해 상속하기보다 조합을 우선 검토합니다.

## TDA: Tell, Don’t Ask

객체의 데이터를 꺼내 판단하지 말고 객체에게 행동을 시킵니다.

나쁜 예:

```java
if (account.getBalance().compareTo(amount) < 0) {
    throw new IllegalStateException();
}
account.setBalance(account.getBalance().subtract(amount));
```

좋은 예:

```java
account.withdraw(amount);
```

효과:

- 캡슐화 강화
- 응집도 증가
- 변경 영향 감소

## SOLID

| 원칙 | 의미 |
| --- | --- |
| SRP | 하나의 변경 이유만 가져야 합니다. |
| OCP | 확장에는 열려 있고 변경에는 닫혀 있어야 합니다. |
| LSP | 하위 타입은 상위 타입을 대체할 수 있어야 합니다. |
| ISP | 클라이언트가 쓰지 않는 인터페이스에 의존하지 않아야 합니다. |
| DIP | 상위 정책은 구체 구현이 아니라 추상화에 의존해야 합니다. |

면접 포인트:

- OCP는 미래를 모두 예측하라는 뜻이 아닙니다.
- 반복되는 변경을 경험한 뒤 추상화로 보호하는 것이 현실적입니다.

## 추상 클래스 vs 인터페이스

| 구분 | 추상 클래스 | 인터페이스 |
| --- | --- | --- |
| 목적 | 공통 상태/로직 재사용 | 역할/계약 정의 |
| 상태 | 인스턴스 필드 가능 | 상수 중심 |
| 다중 구현 | 불가 | 가능 |
| 관계 | is-a | can-do |

## 오버로딩 vs 오버라이딩

| 구분 | 오버로딩 | 오버라이딩 |
| --- | --- | --- |
| 위치 | 같은 클래스 안 | 상속 관계 |
| 기준 | 매개변수 타입/개수/순서 | 같은 시그니처 |
| 결정 시점 | 컴파일 타임 | 런타임 다형성 |
| 목적 | 같은 이름의 다양한 입력 | 상위 동작 재정의 |

## 응집도와 결합도

| 개념 | 좋은 설계 |
| --- | --- |
| 응집도 | 한 모듈의 책임이 서로 강하게 관련됨 |
| 결합도 | 다른 모듈에 대해 아는 것이 적음 |

좋은 방향:

- 높은 응집도
- 낮은 결합도
- 명확한 책임
- 캡슐화된 변경 지점

실무 포인트:

- 테스트 코드를 구현 코드의 첫 사용자로 보면 좋습니다. 테스트 작성이 어렵다는 것은 설계 문제의 신호이며, 주된 관심사와 무관한 코드가 테스트에 많이 개입할수록 설계 개선이 필요하다는 뜻입니다. (출처: 카카오페이)
- SRP는 책임을 변화 관점으로 본 원칙입니다. HTTP 통신과 비즈니스 로직을 한 클래스가 함께 담당하면 API 포맷 변경 시 수정 범위가 넓어지므로 분리하는 것이 좋습니다. (출처: 카카오페이)
- 외부 라이브러리 타입을 반환값으로 노출하면 그 의존성이 모듈 전체로 전파됩니다. 반환 타입을 언어 표준 타입으로 낮추면 HTTP 클라이언트 라이브러리를 교체해도 사용하는 측에 변경이 파급되지 않습니다. (출처: 카카오페이)
- 인프라 조회와 복잡한 비즈니스 로직이 한 서비스 객체에 집중되면 테스트 Given 절 구성이 매우 어려워집니다. 비즈니스 로직만 담당하는 순수 POJO로 분리하면 외부 의존성 없이 객체 생성만으로 복잡한 케이스를 단위 테스트할 수 있습니다. (출처: 카카오페이)
- 결합도를 낮추는 수단으로 헥사고널(port-adapter) 구조를 쓸 수 있습니다. 모듈 간에는 인터페이스(port)만 노출하고 실제 구현(adapter)은 각 도메인 내부에 두며, 도메인 간에는 엔터티가 아니라 DTO·인터페이스로 데이터를 전달합니다. (출처: 우아한형제들)
- 코어 서비스 외부로 엔터티를 직접 반환하지 않는 것을 규칙으로 삼을 수 있습니다. 엔터티가 외부로 노출되면 받은 쪽에서 도메인 로직을 처리하게 되어 로직이 분산되고 서비스 간 결합도가 높아집니다. (출처: 우아한형제들)
- 의존성 방향을 코드로 강제하면 구조가 무너지지 않습니다. 서비스 모듈에서 port 인터페이스를 거쳐 도메인 모듈로 향하는 방향만 허용하고, 이 규칙을 ArchUnit 테스트로 검증해 순환·역방향 의존을 차단할 수 있습니다. (출처: 우아한형제들)
- 트레이드오프도 함께 봐야 합니다. 책임을 나누면 DTO 변환·port 정의로 코드량이 늘고 소규모 변경에도 설계 결정 비용이 커지지만, 인프라 교체 시 비즈니스 로직 무영향과 변경 영향 범위 축소라는 이득이 더 큽니다. (출처: 우아한형제들)

## DI

의존성 주입은 필요한 객체를 내부에서 직접 만들지 않고 외부에서 받는 방식입니다.

| 방식 | 특징 |
| --- | --- |
| 생성자 주입 | 필수 의존성에 적합, 불변성 좋음 |
| Setter 주입 | 선택 의존성에 사용 가능 |
| Method 주입 | 호출마다 달라지는 일시 의존성 |

일반적으로 필수 의존성은 생성자 주입을 선호합니다.

실무 포인트:

- 협력 객체를 외부에서 갈아끼워 격리하면 단위 테스트가 가능해집니다. 테스트 불가능한 영역(예: 미구성 인프라)이 직·간접 의존 관계를 타고 전이되면 결국 전체 코드가 테스트 불가 상태가 되므로, 그 전이를 격리해 나머지 영역의 테스트 가능성을 보존해야 합니다. (출처: 카카오페이)
- Mock 전략은 상황에 따라 선택하는 문제입니다. 실제 환경에 가장 가까운 Mock 서버, 스프링 컨텍스트 재사용을 깨뜨려 빌드가 느려지는 Bean 교체, 멀티 모듈에서 공유 가능한 test-fixtures 사이에서 실제 환경 근접도와 빌드 속도를 저울질합니다. (출처: 카카오페이)
- 테스트를 쉽게 하려고 운영 코드 설계를 훼손하지 않아야 합니다. 테스트용 Mock 설정을 운영 코드 경로에 두면 접근성은 좋아지지만 Mock이 실수로 운영 환경에 올라가는 장애 위험이 생기므로, 테스트 전용 영역으로 명시적으로 분리해야 합니다. (출처: 카카오페이)

## 디자인 패턴

| 패턴 | 핵심 |
| --- | --- |
| Singleton | 인스턴스를 하나만 유지 |
| Strategy | 행위를 인터페이스로 분리해 교체 |
| Template Method | 알고리즘 골격은 상위, 세부 단계는 하위 |
| Null Object | null 대신 아무 일도 하지 않는 객체 사용 |
| First-Class Collection | 컬렉션을 객체로 감싸 관련 로직을 모음 |

### Singleton 주의점

- 전역 상태가 됩니다.
- 테스트 격리가 어렵습니다.
- mutable 상태가 있으면 동시성 문제가 생깁니다.

### Strategy Pattern

변하는 알고리즘을 인터페이스로 분리합니다.

```java
interface MoveStrategy {
    boolean movable(int value);
}
```

효과:

- 조건문 감소
- 알고리즘 교체 쉬움
- OCP 달성에 유리

실무 포인트:

- 템플릿 메서드 패턴에서 상위 클래스가 뼈대에서 많이 기술할수록 하위 클래스의 자유도는 줄어듭니다. 이 제약을 단점이 아니라 모순된 상태 조합을 원천 차단하는 장치로 활용할 수 있습니다. (출처: LINE)
- 상태 간 의존을 상위 클래스에서 강제하면 하위 구현이 순서를 뒤집지 못합니다. 예를 들어 선택 가능 상태가 활성 상태일 때만 성립하도록 계약을 걸면 "활성이 아닌데 선택됨" 같은 모순 상태가 애초에 만들어질 수 없습니다. (출처: LINE)
- 전략 패턴과 템플릿 메서드의 선택 기준은 명확합니다. 전략 패턴은 합성 기반으로 런타임에 알고리즘 전체를 교체할 때 쓰고, 템플릿 메서드는 상속 기반으로 알고리즘의 특정 단계만 제어할 때 씁니다. (출처: LINE)
- 공통 추상 타입에 대해 팩토리에서 일괄 호출만 하면 각 하위 구현이 자신의 단계 메서드를 오버라이딩해 다르게 동작하므로, 클라이언트에서 조건 분기를 제거할 수 있습니다. 이때 저수준이 고수준을 직접 호출하지 않는 할리우드 원칙으로 의존 흐름을 정리합니다. (출처: LINE)
- 트레이드오프도 함께 봐야 합니다. 공통 로직 중복 제거와 계약 강제라는 이득 대신, 하위 구현자가 상위 클래스 코드를 반드시 이해해야 하고 상속이 깊어지면 결합도가 커집니다. (출처: LINE)

## 방어적 복사

가변 객체를 그대로 보관·반환하지 않고 복사해 내부 상태를 보호합니다.

사용 시점:

- 생성자에서 외부 컬렉션을 받을 때
- getter에서 내부 컬렉션을 반환할 때

주의:

- 컬렉션만 복사하면 원소 객체는 공유될 수 있습니다.
- 검증 전에 복사해야 외부 변경 타이밍 문제를 줄일 수 있습니다.

## 함수형 프로그래밍

핵심:

- 순수 함수
- 부수효과 분리
- 함수 합성
- 불변 데이터

객체지향과 함수형은 대립만 하는 개념이 아닙니다. 객체는 변경 지점을 캡슐화하고, 함수형은 부수효과를 줄여 예측 가능성을 높입니다.

## 실전 면접 Q&A

### 캡슐화와 추상화는 어떻게 다른가요?

캡슐화는 변경 가능성을 객체 내부로 숨겨 외부 영향 범위를 줄이는 것이고, 추상화는 중요한 역할과 계약만 드러내고 세부 구현을 감추는 것입니다. 캡슐화는 "무엇을 숨기느냐", 추상화는 "무엇을 드러내느냐"에 초점이 있습니다.

### 상속을 쓸 때 무엇을 조심해야 하나요?

상속은 상위 타입의 속성과 동작을 재사용·확장하지만 강한 결합을 만들 수 있습니다. 재사용만을 목적으로 상속하기보다 조합을 우선 검토하는 것이 좋습니다.

### TDA 원칙이 무엇인가요?

Tell, Don't Ask는 객체의 데이터를 꺼내 판단하지 말고 객체에게 행동을 시키라는 원칙입니다. 잔액을 꺼내 비교하고 다시 설정하는 대신 객체에 출금을 위임하면 캡슐화가 강화되고 응집도가 오르며 변경 영향이 줄어듭니다.

### OCP는 미래의 모든 변경을 예측하라는 뜻인가요?

아닙니다. OCP는 확장에는 열려 있고 변경에는 닫혀 있어야 한다는 원칙이지만, 모든 변화를 미리 예측하라는 의미는 아닙니다. 반복되는 변경을 경험한 뒤 그 지점을 추상화로 보호하는 것이 현실적입니다.

### 추상 클래스와 인터페이스 중 무엇을 고르겠습니까?

공통 상태나 로직을 재사용해야 하면 추상 클래스를 고릅니다. 인스턴스 필드를 가질 수 있고 is-a 관계를 표현합니다. 역할과 계약만 정의하고 다중 구현이 필요하면 인터페이스를 고릅니다. can-do 관계에 적합합니다.

### 오버로딩과 오버라이딩의 차이는 무엇인가요?

오버로딩은 같은 클래스 안에서 매개변수 타입·개수·순서를 달리해 같은 이름의 메서드를 여러 개 두는 것으로 컴파일 타임에 결정됩니다. 오버라이딩은 상속 관계에서 같은 시그니처로 상위 동작을 재정의하는 것으로 런타임 다형성으로 동작합니다.

### 의존성 주입 방식 중 무엇을 선호하나요?

일반적으로 필수 의존성은 생성자 주입을 선호합니다. 불변성을 지키기 좋기 때문입니다. 선택 의존성은 Setter 주입, 호출마다 달라지는 일시 의존성은 Method 주입을 사용할 수 있습니다.

### Singleton 패턴의 단점은 무엇인가요?

인스턴스를 하나만 유지하지만 전역 상태가 되어 테스트 격리가 어렵습니다. 또한 mutable 상태가 있으면 동시성 문제가 생길 수 있어 주의해야 합니다.
