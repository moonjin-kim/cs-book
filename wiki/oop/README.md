# 객체지향

객체지향 면접은 **객체에 책임을 나누고, 변경에 강한 구조를 만들기 위해 캡슐화·추상화·다형성·의존성 관리를 어떻게 사용하는지**를 설명하는 것이 핵심입니다.

## 빠른 요약

| 주제 | 핵심 답변 |
| --- | --- |
| 캡슐화 | 변경 가능성을 객체 내부로 숨겨 외부 영향 범위를 줄입니다. |
| 추상화 | 중요한 역할과 계약만 드러내고 세부 구현을 감춥니다. |
| 다형성 | 같은 메시지를 객체마다 다른 방식으로 처리합니다. |
| SOLID | 변경 이유와 의존성을 관리하기 위한 설계 원칙입니다. |
| DI | 필요한 의존 객체를 외부에서 받아 결합도를 낮춥니다. |
| 응집도/결합도 | 관련 책임은 모으고, 모듈 간 의존은 줄입니다. |
| 디자인 패턴 | 반복되는 설계 문제의 이름 붙은 해법입니다. |

## OOP 4대 특징

| 특징 | 의미 |
| --- | --- |
| 캡슐화 | 데이터와 행위를 묶고 내부 구현을 숨김 |
| 추상화 | 핵심 개념과 계약만 드러냄 |
| 다형성 | 같은 인터페이스가 여러 구현으로 동작 |
| 상속 | 상위 타입의 속성과 동작을 재사용/확장 |

주의:

- 상속은 강한 결합을 만들 수 있습니다.
- 재사용만을 위해 상속하기보다 조합을 우선 검토합니다.

## TDA: Tell, Don’t Ask

객체의 데이터를 꺼내 판단하지 말고 객체에게 행동을 시킵니다.

나쁜 예:

```java
if (account.getBalance().compareTo(amount) < 0) {
    throw new IllegalStateException();
}
account.setBalance(account.getBalance().subtract(amount));
```

좋은 예:

```java
account.withdraw(amount);
```

효과:

- 캡슐화 강화
- 응집도 증가
- 변경 영향 감소

## SOLID

| 원칙 | 의미 |
| --- | --- |
| SRP | 하나의 변경 이유만 가져야 합니다. |
| OCP | 확장에는 열려 있고 변경에는 닫혀 있어야 합니다. |
| LSP | 하위 타입은 상위 타입을 대체할 수 있어야 합니다. |
| ISP | 클라이언트가 쓰지 않는 인터페이스에 의존하지 않아야 합니다. |
| DIP | 상위 정책은 구체 구현이 아니라 추상화에 의존해야 합니다. |

면접 포인트:

- OCP는 미래를 모두 예측하라는 뜻이 아닙니다.
- 반복되는 변경을 경험한 뒤 추상화로 보호하는 것이 현실적입니다.

## 추상 클래스 vs 인터페이스

| 구분 | 추상 클래스 | 인터페이스 |
| --- | --- | --- |
| 목적 | 공통 상태/로직 재사용 | 역할/계약 정의 |
| 상태 | 인스턴스 필드 가능 | 상수 중심 |
| 다중 구현 | 불가 | 가능 |
| 관계 | is-a | can-do |

## 오버로딩 vs 오버라이딩

| 구분 | 오버로딩 | 오버라이딩 |
| --- | --- | --- |
| 위치 | 같은 클래스 안 | 상속 관계 |
| 기준 | 매개변수 타입/개수/순서 | 같은 시그니처 |
| 결정 시점 | 컴파일 타임 | 런타임 다형성 |
| 목적 | 같은 이름의 다양한 입력 | 상위 동작 재정의 |

## 응집도와 결합도

| 개념 | 좋은 설계 |
| --- | --- |
| 응집도 | 한 모듈의 책임이 서로 강하게 관련됨 |
| 결합도 | 다른 모듈에 대해 아는 것이 적음 |

좋은 방향:

- 높은 응집도
- 낮은 결합도
- 명확한 책임
- 캡슐화된 변경 지점

## DI

의존성 주입은 필요한 객체를 내부에서 직접 만들지 않고 외부에서 받는 방식입니다.

| 방식 | 특징 |
| --- | --- |
| 생성자 주입 | 필수 의존성에 적합, 불변성 좋음 |
| Setter 주입 | 선택 의존성에 사용 가능 |
| Method 주입 | 호출마다 달라지는 일시 의존성 |

일반적으로 필수 의존성은 생성자 주입을 선호합니다.

## 디자인 패턴

| 패턴 | 핵심 |
| --- | --- |
| Singleton | 인스턴스를 하나만 유지 |
| Strategy | 행위를 인터페이스로 분리해 교체 |
| Template Method | 알고리즘 골격은 상위, 세부 단계는 하위 |
| Null Object | null 대신 아무 일도 하지 않는 객체 사용 |
| First-Class Collection | 컬렉션을 객체로 감싸 관련 로직을 모음 |

### Singleton 주의점

- 전역 상태가 됩니다.
- 테스트 격리가 어렵습니다.
- mutable 상태가 있으면 동시성 문제가 생깁니다.

### Strategy Pattern

변하는 알고리즘을 인터페이스로 분리합니다.

```java
interface MoveStrategy {
    boolean movable(int value);
}
```

효과:

- 조건문 감소
- 알고리즘 교체 쉬움
- OCP 달성에 유리

## 방어적 복사

가변 객체를 그대로 보관하거나 반환하지 않고 복사해 내부 상태를 보호합니다.

사용 시점:

- 생성자에서 외부 컬렉션을 받을 때
- getter에서 내부 컬렉션을 반환할 때

주의:

- 컬렉션만 복사하면 원소 객체는 공유될 수 있습니다.
- 검증 전에 복사해야 외부 변경 타이밍 문제를 줄일 수 있습니다.

## 함수형 프로그래밍

핵심:

- 순수 함수
- 부수효과 분리
- 함수 합성
- 불변 데이터

객체지향과 함수형은 대립만 하는 개념이 아닙니다. 객체는 변경 지점을 캡슐화하고, 함수형은 부수효과를 줄여 예측 가능성을 높입니다.
