# Operating System

OS는 애플리케이션이 CPU, 메모리, 디스크, 네트워크를 직접 다루지 않아도 되게 만드는 추상화 계층입니다.

## 핵심 개념

| 개념 | 알아야 할 질문 |
| --- | --- |
| Process | 주소 공간, 파일 디스크립터, 권한, 상태는 무엇을 의미하는가? |
| Thread | 프로세스와 무엇을 공유하고 무엇을 별도로 가지는가? |
| Scheduling | 선점/비선점, FCFS, SJF, RR, SRT, MLFQ는 어떤 trade-off가 있는가? |
| Context Switching | 프로세스 전환이 스레드 전환보다 비싼 이유는 무엇인가? |
| System Call | 사용자 모드 프로그램은 어떻게 커널 서비스를 요청하는가? |
| 동시성/병렬성 | 논리적으로 동시에 보이는 것과 물리적으로 동시에 실행되는 것은 어떻게 다른가? |
| Deadlock | 교착 상태의 4가지 조건과 회피 방법은 무엇인가? |
| Memory | 연속 할당, 페이징, 가상 메모리, 페이지 교체는 어떻게 연결되는가? |
| Disk/I/O | seek time, rotational latency, transfer time은 디스크 성능을 어떻게 결정하는가? |

## 단일 프로세스, 멀티프로그래밍, 멀티태스킹

단일 프로세스 시스템은 한 번에 하나의 프로그램만 실행합니다. 실행 중인 프로그램이 I/O를 기다리면 CPU가 놀게 되므로 CPU 사용률이 낮습니다.

멀티프로그래밍은 한 프로세스가 I/O로 대기할 때 다른 프로세스를 실행해 CPU 유휴 시간을 줄입니다. 목적은 CPU 사용률 극대화입니다. 다만 CPU-bound 프로세스가 CPU를 오래 점유하면 다른 프로세스가 기다리는 문제가 남습니다.

멀티태스킹은 CPU 시간을 짧은 quantum으로 나누어 여러 프로세스가 번갈아 실행되게 합니다. 목적은 응답 시간 개선이며, 사용자는 여러 프로그램이 동시에 실행되는 것처럼 느낍니다.

## 프로세스와 스레드

멀티태스킹만으로는 하나의 프로세스 안에서 여러 작업을 동시에 처리하기 어렵고, 프로세스 간 컨텍스트 스위칭과 데이터 공유 비용도 큽니다. 이 한계를 줄이기 위해 스레드가 사용됩니다.

- 프로세스는 독립된 주소 공간과 자원을 가진 실행 중인 프로그램입니다.
- 스레드는 프로세스 안에서 CPU 스케줄링을 받는 실행 단위입니다.
- 같은 프로세스의 스레드들은 heap 같은 메모리 영역을 공유하지만, 각자 stack, program counter, register 상태를 가집니다.
- 멀티스레딩은 하나의 프로세스 안에서 여러 스레드로 작업을 나누는 방식입니다.
- 멀티프로세싱은 둘 이상의 CPU나 코어를 활용해 여러 프로세스 또는 실행 흐름을 병렬 처리하는 방식입니다.

스레드, 프로세스, 코어는 많을수록 무조건 좋은 것이 아닙니다. 스레드가 너무 많으면 컨텍스트 스위칭, 락 경쟁, 캐시 미스, 메모리 사용량이 늘고, 프로세스가 많으면 독립 주소 공간과 IPC 비용이 커집니다. 코어가 많아도 소프트웨어가 병렬화되어 있지 않으면 이점을 얻기 어렵습니다.

## Context Switching

컨텍스트 스위칭은 CPU에서 실행 중이던 프로세스나 스레드의 상태를 저장하고, 다음 실행 대상의 상태를 복원하는 과정입니다. 보통 time slice 소진, I/O 대기, 더 높은 우선순위 작업 등장 시 발생합니다.

프로세스 컨텍스트 스위칭은 다른 주소 공간으로 전환해야 하므로 MMU 설정과 TLB 처리 같은 메모리 주소 관련 비용이 추가됩니다. 반면 같은 프로세스 내부의 스레드 전환은 주소 공간을 공유하므로 주로 레지스터, stack pointer, program counter 같은 CPU 실행 상태를 바꾸면 됩니다. 그래서 일반적으로 스레드 전환이 프로세스 전환보다 가볍습니다.

## System Call

운영체제는 응용 프로그램이 하드웨어 자원에 직접 접근하지 못하게 보호합니다. 일반 프로그램은 사용자 모드에서 실행되며, 파일 I/O, 네트워크, 프로세스 생성 같은 커널 서비스가 필요할 때 시스템 콜을 호출합니다.

시스템 콜은 사용자 모드에서 커널 모드로 전환하기 위한 소프트웨어 인터럽트입니다. CPU는 현재 실행 상태를 저장하고 커널의 시스템 콜 처리 루틴을 실행한 뒤, 다시 원래 프로그램으로 복귀합니다.

## 동시성과 병렬성

동시성은 여러 작업이 논리적으로 동시에 진행되는 것처럼 보이게 만드는 개념입니다. 단일 코어에서도 time slicing과 context switching으로 여러 스레드를 번갈아 실행하면 동시성을 얻을 수 있습니다. I/O 대기와 응답성 개선에 유리하지만 race condition, deadlock, starvation 같은 문제가 생길 수 있습니다.

병렬성은 여러 코어가 실제 같은 시간에 여러 작업을 실행하는 것입니다. 독립적인 계산 작업을 여러 코어에 나누면 처리 시간이 줄지만, 공유 데이터가 있으면 동기화 비용이 발생합니다.

## Race Condition과 Deadlock

Race condition은 여러 스레드가 공유 자원에 동시에 접근할 때 실행 순서에 따라 결과가 달라지는 현상입니다. 해결하려면 원자성과 가시성이 함께 보장되어야 합니다.

- 원자성: `i++` 같은 read-modify-write 연산이 중간에 끼어들 수 없는 하나의 연산처럼 수행되어야 합니다.
- 가시성: 한 스레드의 변경이 다른 스레드에서 관찰 가능해야 합니다.

Deadlock은 두 개 이상의 작업이 서로 상대가 가진 자원을 기다려 아무것도 진행하지 못하는 상태입니다.

| 조건 | 설명 |
| --- | --- |
| Mutual Exclusion | 한 자원을 동시에 여러 작업이 사용할 수 없음 |
| Hold and Wait | 자원을 가진 채 다른 자원을 기다림 |
| Non-preemption | 점유한 자원을 강제로 빼앗을 수 없음 |
| Circular Wait | 작업들이 원형으로 서로의 자원을 기다림 |

해결은 네 조건 중 하나를 깨는 방식입니다. lock 획득 순서를 일관되게 하거나, 중첩 lock을 줄이거나, `tryLock(timeout)`, `lockInterruptibly()`로 무한 대기를 피합니다.

## CPU Scheduling

CPU 스케줄링은 운영체제가 준비 상태의 프로세스나 스레드에 CPU 시간을 배분하는 정책입니다.

| 방식 | 설명 | 주의점 |
| --- | --- | --- |
| FCFS | 준비 큐에 들어온 순서대로 실행하는 비선점형 방식 | 긴 작업이 앞에 있으면 convoy effect 발생 |
| SJF | 실행 시간이 가장 짧은 작업을 먼저 실행하는 비선점형 방식 | 긴 작업의 starvation 가능 |
| RR | time slice만큼 돌아가며 실행하는 선점형 방식 | quantum이 너무 작으면 문맥 교환 비용 증가 |
| SRT | SJF의 선점형 버전. 남은 시간이 더 짧은 작업이 오면 선점 | 실행 시간 추정이 필요하고 starvation 가능 |
| Multilevel Queue | 우선순위별 큐를 나누어 처리 | 낮은 우선순위 큐의 starvation 가능 |
| MLFQ | 오래 기다린 작업의 우선순위를 올리는 aging을 적용 | 설정이 복잡하지만 starvation 완화 |

선점형 스케줄링은 응답성이 좋지만 컨텍스트 스위칭 오버헤드와 경쟁 상태 관리가 필요합니다. 비선점형 스케줄링은 단순하고 전환 비용이 적지만 응답 시간이 길어질 수 있습니다.

## 연속 메모리 할당과 단편화

연속 메모리 할당은 프로세스에 연속적인 물리 메모리 공간을 할당하는 방식입니다.

- 고정 크기 할당: 메모리를 고정 크기 블록으로 나누어 할당합니다. 필요한 크기보다 큰 블록을 받으면 내부 단편화가 발생합니다.
- 가변 크기 할당: 프로세스 크기에 맞춰 할당합니다. 할당과 해제가 반복되면 작은 빈 공간이 흩어져 외부 단편화가 발생합니다.

빈 공간에 프로세스를 배치하는 대표 전략은 최초 적합, 최적 적합, 최악 적합입니다.

| 전략 | 설명 |
| --- | --- |
| First Fit | 처음 발견한 충분한 빈 공간에 배치 |
| Best Fit | 들어갈 수 있는 가장 작은 빈 공간에 배치 |
| Worst Fit | 들어갈 수 있는 가장 큰 빈 공간에 배치 |

## 가상 메모리와 페이지 교체

가상 메모리는 프로세스의 논리 주소 공간을 물리 메모리보다 크게 사용할 수 있게 하는 기법입니다. 페이징은 논리 주소 공간을 page로, 물리 메모리를 frame으로 나누어 매핑합니다.

요구 페이징은 실제로 필요한 page만 메모리에 적재합니다. 새 page를 올려야 하는데 frame이 부족하면 기존 page 중 하나를 내보내야 하며, 이 선택 기준이 페이지 교체 알고리즘입니다.

| 알고리즘 | 설명 |
| --- | --- |
| FIFO | 가장 먼저 적재된 page를 교체 |
| OPT | 앞으로 가장 늦게 사용될 page를 교체하는 이론적 최적 알고리즘 |
| LRU | 가장 오래 사용되지 않은 page를 교체 |
| LFU | 사용 빈도가 가장 낮은 page를 교체 |
| NUR | reference bit와 modified bit를 활용해 최근 사용되지 않은 page를 교체 |

LRU, LFU, NUR, FIFO 변형은 과거 접근 패턴으로 미래를 추정하는 최적 근접 알고리즘입니다.

## 디스크 접근 시간

하드 디스크에서 특정 블록을 읽거나 쓰려면 헤드를 트랙으로 이동하고, 원하는 섹터가 헤드 아래로 올 때까지 기다린 뒤, 데이터를 전송해야 합니다.

디스크 접근 시간은 다음 요소의 합으로 볼 수 있습니다.

- Seek Time: 헤드를 필요한 트랙으로 이동하는 시간
- Rotational Latency: 원하는 섹터가 회전해 올 때까지 기다리는 시간
- Data Transfer Time: 실제 데이터를 전송하는 시간

랜덤 액세스는 여러 위치로 헤드를 이동하고 매번 회전 지연을 기다려야 하므로 느립니다. 순차 액세스는 기계적 이동이 적어 훨씬 빠릅니다.

## 실무 판단

- 서버 장애 분석은 CPU, memory, disk I/O, network I/O를 분리해서 봅니다.
- thread 수를 늘리는 것은 I/O 대기에는 도움이 될 수 있지만 CPU 병목에는 역효과가 날 수 있습니다.
- 컨테이너는 OS 커널을 공유하는 격리된 프로세스라는 점을 기억해야 합니다.
- 디스크 성능 문제는 랜덤 I/O인지 순차 I/O인지부터 확인합니다.
