# 객체지향

객체지향은 클래스를 많이 만드는 기술이 아니라 변경 이유를 적절한 책임 경계 안에 가두는 설계 방식입니다.

## 핵심 개념

| 개념 | 알아야 할 질문 |
| --- | --- |
| 캡슐화 | 어떤 변경을 객체 내부로 숨기고 어떤 계약만 외부에 공개할 것인가? |
| 추상화 | 구체 구현이 아니라 어떤 역할과 메시지에 의존할 것인가? |
| 다형성 | 조건문을 객체 협력으로 바꿀 수 있는 지점은 어디인가? |
| 의존성 역전 | 고수준 정책이 저수준 구현에 끌려가지 않게 하려면 무엇이 필요한가? |
| 디자인 패턴 | 패턴이 해결하는 문제와 새로 만드는 복잡도는 무엇인가? |

## 실무 판단

- 상속은 재사용보다 확장 지점이 명확할 때 선택합니다.
- 인터페이스는 "테스트를 위해서"보다 "실제로 바뀔 수 있는 정책"에 둡니다.
- SOLID는 체크리스트가 아니라 변경 비용을 낮추기 위한 관찰 도구입니다.

## OOP와 SOLID

객체지향 프로그래밍은 상태와 행위를 가진 객체에 역할과 책임을 부여하고, 객체들이 메시지를 주고받으며 협력하게 만드는 패러다임입니다. 핵심 특징은 캡슐화, 추상화, 다형성, 상속입니다.

캡슐화는 내부 구현과 변경 가능성을 객체 안에 숨기고 외부에는 필요한 계약만 공개합니다. TDA(Tell, Don't Ask)는 객체의 데이터를 꺼내 판단하지 말고 객체에게 일을 시키라는 원칙으로, 캡슐화와 응집도 유지에 도움이 됩니다.

SOLID는 변경 비용을 낮추기 위한 설계 원칙입니다.

| 원칙 | 핵심 |
| --- | --- |
| SRP | 클래스는 하나의 변경 이유만 가져야 함 |
| OCP | 확장에는 열려 있고 기존 코드 변경에는 닫혀 있어야 함 |
| LSP | 하위 타입은 상위 타입의 계약을 깨지 않아야 함 |
| ISP | 클라이언트가 쓰지 않는 interface에 의존하지 않게 분리 |
| DIP | 고수준 정책과 저수준 구현 모두 추상화에 의존 |

OCP는 미래를 완벽히 예측하라는 뜻이 아닙니다. 변화가 실제로 반복될 때 그 변화 축을 추상화해 다음 변경 비용을 낮추는 쪽에 가깝습니다.

## 응집도와 결합도

응집도는 모듈 내부 요소들이 같은 책임을 위해 얼마나 밀접하게 관련되는지, 결합도는 다른 모듈을 얼마나 많이 알고 의존하는지를 뜻합니다. 좋은 설계는 일반적으로 높은 응집도와 낮은 결합도를 가집니다.

캡슐화가 약하면 내부 상태와 구현 세부사항이 외부로 새어 나가 결합도가 높아집니다. 반대로 객체가 자신의 데이터를 스스로 다루고 필요한 메시지만 공개하면 응집도는 높아지고 변경 전파는 줄어듭니다.

## 의존성 주입

의존성 주입은 객체가 필요한 의존 객체를 직접 생성하지 않고 외부에서 전달받는 방식입니다. 생성 책임과 사용 책임을 분리하므로 구현체 교체, 테스트 대역 주입, 정책 변경이 쉬워집니다.

생성자 주입은 필수 의존성을 불변으로 보장하기 좋고, setter 주입은 선택 의존성이나 실행 중 교체가 필요할 때 쓸 수 있습니다. 메서드 주입은 특정 호출에서만 필요한 일시적 의존에 적합합니다.

## 디자인 패턴

Singleton은 하나의 instance를 전역적으로 공유하는 패턴입니다. 메모리 절약과 접근 편의성이 있지만, 전역 상태로 인해 테스트 격리와 동시성 문제가 생기기 쉽습니다. Spring singleton bean도 공유 상태를 두면 같은 위험을 가집니다.

Strategy는 바뀌는 행위를 interface로 분리하고 구현체를 주입해 알고리즘을 교체하는 패턴입니다. 조건문으로 여러 정책을 나누는 대신 다형성으로 확장할 수 있습니다.

Template Method는 알고리즘의 골격은 상위 클래스에 두고 세부 단계는 하위 클래스가 구현하게 합니다. 중복을 줄일 수 있지만 상속 결합이 강하고 상위 클래스 변경이 하위 클래스에 영향을 크게 줄 수 있습니다.

Null Object는 null 대신 아무 일도 하지 않는 객체를 전달해 반복 null check를 줄입니다. 다만 오류를 조용히 숨길 수 있으므로 "없음"이 정상 도메인 개념일 때 신중하게 사용합니다.

## 일급 컬렉션과 방어적 복사

일급 컬렉션은 collection 하나를 감싼 전용 class입니다. 컬렉션 관련 검증, 계산, 변경 규칙을 한 곳에 모으고, 불필요한 `add`, `remove`, `set` 같은 API 노출을 제한할 수 있습니다.

방어적 복사는 생성자 인자나 getter 반환값의 참조 공유를 끊기 위해 복사본을 만드는 방식입니다. 생성자에서는 먼저 복사한 뒤 복사본을 검증해야 검증과 대입 사이 외부 변경을 막을 수 있습니다. 단순 `new ArrayList<>(list)`는 얕은 복사이므로 내부 원소가 가변 객체라면 깊은 복사나 불변 객체 설계가 필요합니다.

## 함수형 프로그래밍

함수형 프로그래밍은 부수 효과를 줄이고 순수 함수의 합성으로 프로그램을 구성하려는 패러다임입니다. 순수 함수는 같은 입력에 항상 같은 출력을 반환하고 외부 상태를 변경하지 않습니다. 부수 효과가 적으면 테스트와 추론이 쉬워지고, 함수 합성으로 복잡한 처리를 작은 변환들의 연결로 표현할 수 있습니다.

객체지향과 함수형은 배타적이지 않습니다. Java에서도 domain 객체는 책임과 협력으로 설계하고, collection 변환이나 계산 로직은 Stream, 함수형 interface, 불변 객체를 활용해 함수형 스타일로 작성할 수 있습니다.
