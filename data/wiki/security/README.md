# Security

보안 면접은 **인증과 인가를 구분하고, 전송·저장·입력 처리에서 어떤 위협을 막아야 하는지**를 설명하는 것이 핵심입니다.

## 빠른 요약

| 주제 | 핵심 답변 |
| --- | --- |
| 인증 | 사용자가 누구인지 확인합니다. |
| 인가 | 인증된 사용자가 무엇을 할 수 있는지 판단합니다. |
| JWT | 서명된 claim token이며 탈취, 만료, key 관리가 중요합니다. |
| HTTPS | TLS로 서버 인증, 키 교환, 암호화를 수행합니다. |
| Hash | 비밀번호는 salt와 느린 단방향 해시로 저장합니다. |
| SQL Injection | 입력값이 SQL 구조를 바꾸는 공격입니다. |
| CSRF | 로그인 상태를 악용해 원치 않는 요청을 보내게 하는 공격입니다. |

## 인증과 인가

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| 인증 Authentication | 사용자가 누구인지 확인 | 로그인, MFA |
| 인가 Authorization | 어떤 권한이 있는지 확인 | 관리자 API 접근 |

흐름:

```text
인증 -> principal 식별 -> 권한 조회 -> 인가 판단
```

## Session vs Token

| 구분 | Session | Token/JWT |
| --- | --- | --- |
| 상태 저장 | 서버 저장 | 보통 서버 저장 없음 |
| 확장성 | 세션 저장소 필요 | 서버 간 공유 쉬움 |
| 폐기 | 서버에서 즉시 가능 | 만료 전 폐기 어려움 |
| 보안 포인트 | session id 보호 | token 탈취, 만료, refresh 관리 |

## JWT

JWT 구성:

```text
Header.Payload.Signature
```

| 부분 | 내용 |
| --- | --- |
| Header | 알고리즘, 토큰 타입 |
| Payload | claim |
| Signature | 위변조 검증 |

주의:

- Payload는 암호화가 아니라 Base64URL 인코딩입니다.
- 민감 정보를 넣지 않습니다.
- access token은 짧게, refresh token은 안전하게 보관합니다.
- key rotation 전략이 필요합니다.

## 암호화와 해시

| 구분 | 목적 | 복호화 |
| --- | --- | --- |
| 대칭키 암호화 | 빠른 데이터 암호화 | 가능 |
| 공개키 암호화 | 키 교환, 서명 | 가능 |
| 해시 | 무결성, 비밀번호 저장 | 불가능 |
| 전자서명 | 위변조 방지, 부인 방지 | 검증 |

## HTTPS/TLS

TLS가 제공하는 것:

- 서버 인증
- 키 교환
- 대칭키 암호화
- 메시지 무결성

흐름 요약:

1. 서버 인증서 검증
2. 키 교환
3. 세션 키 생성
4. 대칭키로 HTTP 데이터 암호화

## 비밀번호 저장

비밀번호는 복호화 가능한 암호화가 아니라 단방향 해시로 저장합니다.

필수 요소:

- Salt
- 느린 해시
- 충분한 반복 비용

권장 알고리즘:

- bcrypt
- scrypt
- Argon2

지양:

- 단순 SHA-256
- salt 없는 hash
- 평문 저장

## OWASP 주요 위험

| 위험 | 설명 |
| --- | --- |
| Broken Access Control | 권한 검증 실패 |
| Injection | 입력이 명령이나 쿼리 구조를 변경 |
| XSS | 악성 script 실행 |
| CSRF | 인증 상태를 악용한 요청 위조 |
| SSRF | 서버가 내부망 요청을 대신 보내게 함 |

## SQL Injection

원인:

- 사용자 입력을 SQL 문자열에 직접 이어 붙임

위험:

- 데이터 유출
- 인증 우회
- 데이터 변조/삭제

대응:

- PreparedStatement 사용
- ORM parameter binding
- 입력값 검증
- 최소 권한 DB 계정

## CSRF

CSRF는 사용자의 로그인 상태를 악용해 의도하지 않은 요청을 보내게 합니다.

대응:

- CSRF token
- SameSite cookie
- 중요한 변경 요청에 재인증
- GET으로 상태 변경 금지

## CORS와 보안 오해

CORS는 브라우저의 cross-origin 요청 제한입니다.

주의:

- CORS는 인증/인가 대체 수단이 아닙니다.
- 서버 간 요청은 CORS로 막을 수 없습니다.
- API 권한 검사는 서버에서 반드시 수행해야 합니다.
